## 1. Problem & Motivation

- Document creation and storage has shifted toward git repositories, driven largely by the volume of output that AI-assisted drafting produces.
- Git gives stronger version control and folder discipline than Google Drive, which is valued and worth keeping.
- Git does not natively provide two things Google Drive gave for free:
  - Collaborative live editing
  - Rich, threaded, in-place comments
- Of the two, **comments matter more right now** — AI-assisted workflows have already changed how collaborative editing tends to happen, reducing the urgency of that gap.
- Prior research (separate report, "Rich Commenting on Git-Versioned Documents") surveyed existing tools against this gap. Conclusion: nothing combines git-as-source-of-truth with rich, Google-Docs-style commenting. See Section 3 for the short version.

---

<!-- nog:blk-002 -->
## 2. Target Use Cases

- One or two people, working with an AI, produce documents considered useful enough to commit to the working repository.
- Documents need to be shared with a wider group who are **not necessarily part of the organisation** and are **unlikely to have repo access**.
- Anyone in that wider group may:
  - Comment on the document
  - Respond to comments in a thread
  - Edit commented-on sections manually, and identify their edit as resolving a comment
  - Review the document's history to understand how and why it changed — including how comments were resolved

---

## 3. Why Not an Existing Tool (short version)

- **PR/MR review (GitHub/GitLab):** git-native, but comments are diff-anchored, not anchored to rendered prose, and require the commenter to have a repo account — disqualifying for an external, non-technical audience.
- **md-review-extension:** improves PR review of rendered markdown, but still GitHub-account-gated and comments still live inside a PR.
- **giscus / utterances:** page-level comments on a published site, but backed by GitHub Issues/Discussions — still requires a GitHub account to participate.
- **Hypothesis:** true inline annotation on any published page, lowest-friction account of the bunch, but annotation-only — no editing, no resolution workflow, no tie to git history.
- **HackMD / HedgeDoc:** real-time editing + comments, closest existing match — but git is a satellite via export/sync, not the source of truth, and doesn't fit an audience that shouldn't need repo-adjacent tooling at all.
- **Overleaf + Git:** the cautionary tale — Overleaf's own docs warn that active git use can silently destroy comments/track changes. Direct evidence this is a genuinely unsolved problem, not just a tooling gap.
- **TinaCMS:** solves git-friendly editing, has no comment layer at all.
- **CodeStream:** inline comments anchored to git blame, but IDE-based — built for developers, not external prose reviewers.
- **Moxn:** very new entrant pitched at exactly this space ("git-like version control for collaborative docs"), too early-stage to assess.
- **Conclusion:** none of the surveyed tools satisfy the fuller use-case list in Section 2, mainly because they assume git-platform identity for commenters. This is the gap Nog is meant to fill.

## 4. Working Assumptions / Decisions So Far

<!-- nog:blk-003 -->
### 4.1 Identity & Access Model

- Two distinct populations, with different capabilities:
  - **Core team** — has git repo write access, drafts with AI assistance.
  - **Wider group** — no repo access, no git literacy assumed, may be outside the organisation.
- Nog's permissions and identity are entirely decoupled from git identity. The wider group never touches git directly.
- Permission model is Google-Docs-like: **read / comment / edit / own**, managed at the Nog level, not the git level.
- The repo remains the canonical source of truth. Anything an external person contributes is mediated back into the repo via a PR/MR generated by an authorized Nog user (see 4.4) — never committed directly by external users, and reconciliation of any divergent edits happens downstream in git/agent space, not inside Nog.

<!-- nog:blk-004 -->
### 4.2 Document Representation

- **Front matter** is the home for document metadata (an established, low-risk convention — used by Jekyll/Hugo-style tools for over a decade).
  - Holds a **persistent document UID**, independent of filename — needed because people rename AI-produced documents, and git's rename detection is a similarity heuristic, not a guarantee (especially fragile when a rename and a heavy edit happen together).
  - Because git commit history doesn't know what a comment is, it was never going to serve as the audit trail on its own (see Section 2's history use case). Nog needs its own document-history record, keyed on the UID, using git commits as raw evidence rather than as the index.
- **Block IDs**, not full JSON-ification of the document, are the anchoring mechanism for comments.
  - Rejected approach: parsing the whole document into a JSON/AST structure to make it addressable (à la "Google Docs is a JSON object"). Flagged as the highest-risk part of any earlier design — round-trip fidelity (markdown → JSON → markdown) and anchor stability under concurrent edits is arguably as hard as the original problem.
  - Block IDs are lighter-weight: they tag stable anchor points into otherwise-untouched markdown, so files stay plain markdown, editable directly by a human or an AI with no special tooling.
  - The **structured/addressable representation only needs to exist transiently**, materialized on demand when someone opens a specific document to comment or edit — not maintained as an always-on parallel copy.

<!-- nog:blk-005 -->
### 4.3 Comment & Thread Data Model

- Comments, threads, and intermediate (not-yet-committed) document versions live in a **database external to git** — the right tool for structured, relational, threaded data that git handles poorly.
- Resolution needs more than one state:
  - Resolved via a specific edit (linked: comment → thread → the edit that addressed it → resolver identity)
  - Resolved via discussion alone, with no text change ("acknowledged," "won't fix") — without this state, threads either stay open forever or force an unnecessary edit just to get closure.
- **Comment threads are single-threaded, not nested:** an original comment plus a flat list of replies underneath it, no reply-to-reply trees. Keeps the data model and UI simple, and mirrors how comment threads already work in Google Docs and GitHub PR reviews. As a side effect, replying is always conflict-free — it's an append to a list, never a structural edit — which narrows the "concurrent commenters" concern in 4.4 down to just concurrent editing of the same block.
- **Resolution is recorded as a Nog-space event, not reconstructed later:** closing a comment after making an edit creates an explicit event in Nog's own database at the moment it happens — who closed it, and which edit resolved it. This is what keeps attribution intact regardless of anything that happens later during the eventual git-side merge (see 4.4): Nog already knows "who resolved comment X with edit Y" before any downstream AI-guided merge ever touches the text.
- **Resolved threads are locked, not deleted:** once a comment is resolved — via edit or via discussion, either state above — the thread becomes read-only, no further replies. Distinct from the 4.4 paragraph-level freeze: that's a temporary (~1 minute), *block*-scoped lock that fires during active editing; this is a permanent, *thread*-scoped lock that fires on resolution. Reopening a resolved thread is deliberately out of scope for now — a conscious decision to defer, not an oversight, leaving room to design it properly later rather than bolt it on now.
- **Version drift / orphaned comments:** a comment is anchored to a specific state of the text. If that passage changes or disappears before the comment is addressed, there needs to be an explicit rule — auto-resolve, flag as orphaned, or preserve the original anchored text so later readers can still see what was being discussed.
- The "review history" use case (Section 2) implies a **synthesized narrative** — comment raised → discussion → edit → resolution — not a raw git log. This is a deliberate construction on top of both the comment database and git history, not something either gives you natively.

### 4.4 Resolution Model: Staged in the Database, Fork Deferred Until First Resolved Edit

- **Core idea, substantially revised (see changelog v9):** resolving a comment is not "propose a diff and reconcile it with other diffs" — it's a human with resolve authority looking at whatever text is *currently, singly* live for a paragraph and certifying it acceptable. Comments, discussion, and candidate rewordings all accumulate as database artifacts first; git is touched only once something has actually been resolved. This is a stronger outcome than earlier framings of this section, which treated AI-guided merge as out of Nog's scope — under this model, the scenario AI-guided merge existed to solve (two independently-finished, divergent edits to the same paragraph needing reconciliation) can no longer arise at all. See "what this eliminates," below.
- **Three tiers, only the last of which touches git:**
  - **Ephemeral candidate:** someone selects text and starts drafting a comment or a candidate edit, but never submits. Never persisted server-side; if the page is left or the session ends, it simply vanishes. Exact mechanics (e.g. purely client-side state until submit) are an implementation detail, not a design decision this note needs to pin down.
  - **Submitted comment / candidate edit (database only, no git):** once a comment — and optionally a proposed alternate wording for the paragraph — is actually submitted, it's a real, persistent Nog database artifact, with a thread and replies. Nothing about this touches the repository, and this state can persist indefinitely; a comment with no proposed edit yet is a completely normal, stable state, not a fork waiting to happen.
  - **Resolution — the only trigger that touches git, gated by two explicit steps:** clicking Resolve doesn't commit anything by itself — it shows the resolver exactly what text is about to be certified, and requires a separate, explicit confirmation before anything happens. This is specifically a fat-finger guard, not a loop guard (see below): it protects against someone certifying text they didn't mean to, not against any kind of runaway automation. Only once confirmed does a fork get created (if none already exists for this document, per the single-live-fork rule below) and the resolved text committed, carrying the commit-context trailer (still valuable — see below). The commit itself can be silent — the safety work already happened at the confirmation step, so the write afterward doesn't need its own visible feedback.
  - Maps closely onto git's own staged/unstaged/committed distinction: comments and candidate edits are the working directory, resolution is the commit. Worth keeping that analogy close at hand — it does real explanatory work, not just decorative.
- **What this eliminates:** the earlier design (through v6) assumed two people could independently produce two finished, divergent edits to the same paragraph that would then need AI-guided reconciliation. That can't happen here. The paragraph-level freeze (below) ensures only one candidate edit is ever active on a given block at a time, and resolution certifies whatever the single current state is, rather than reconciling competing proposals. Complex judgment is resolved by a human pressing the button, never automated — the "Reddit approach." This is a stronger result than the Six Thinking Hats review's Blue Hat recommendation to spike-test the AI-merge assumption: the assumption is now unnecessary rather than merely untested. "Nog - Thin-Slice Pilot Outline" (Phase 2) and "Nog - AI Merge Spike Checklist" are marked superseded rather than deleted, preserving the reasoning rather than erasing it.
  - **What survives, reframed:** the commit-message trailer (bounded verbatim excerpt + reference ID) still earns its place — not as context for an AI merge agent, which no longer exists, but as a permanent, git-native audit-trail annotation for the history use case in Section 2. Likewise, Nog compiling a fuller description at PR-generation time still has value — not for an AI, but for whichever human reviews the PR.
  - **What's retired:** the "single AI invocation, once per fork, at PR generation" model is no longer needed.
- **Decided, otherwise unchanged from earlier versions of this section:**
  - **Single live fork per document:** only one fork may be open for a document at a time — now instantiated later (first resolution, not first comment), but the discipline itself is unchanged. A Google Doc only ever has one current instance; too many concurrent versions make a document unusable well before git mechanics become the bottleneck.
  - **Merge-back trigger:** a Nog user with the relevant git repo authorization manually generates a PR/MR from the fork. Since every commit on the fork is already a human-approved resolution by this point, this PR review is close to a formality/final check, not a "resolve competing text" task.
  - **New resolutions arriving while the fork's PR is already pending are treated like unstaged changes** — unchanged from earlier versions, just triggered later in the sequence (by a resolution, not by any comment or edit in isolation). Folding pending items into the current PR, or holding them for the next round, is a deliberate action, not automatic. Nog should visually distinguish the two states (colour-coding, or similar).
- **Editing scope, made explicit:** once a candidate edit is possible on a paragraph, editing is restricted to that one paragraph at a time, on threads with an open comment — never general free-roam editing anywhere in the document. Nog is a commenting and light-resolution tool for a document authored elsewhere, not a reauthoring surface; broader editing capability is explicitly out of scope, sharpening the original framing in Section 1 ("more concerned with rich comments than with collaborative editing").
- **Freeze mechanism, relocated:** the time-boxed, paragraph-level freeze — starting an edit or a new comment locks that block until roughly a minute after posting, with its own separate inactivity timeout for abandoned edits, and never triggered by plain replies — now operates entirely at the database layer, before any fork exists. There's no git working tree being contended over during drafting, since nothing touches git until resolution. The mechanism itself is unchanged from earlier versions of this section; only where it lives is clarified.
- **Not yet resolved by this revision:**
  - **Multi-paragraph resolutions — deliberately parked for MVP, not solved.** One paragraph, one comment is accepted as a deliberately dumb first-version constraint. Adopting an existing block-based editor library for the edit window (ProseMirror-derived tooling and similar are the likely category) will probably make the *editing mechanics* of multi-block selection, splitting, and merging close to free — that capability tends to come with the library. It will not solve Nog's own problem underneath that: what happens to a block ID's identity, and anything anchored to it (comments, a resolved/locked thread, the git-side mapping in 4.2), when the paragraph that ID names gets split or merged. That's specific to Nog's own anchoring layer, not something any borrowed editor knows to care about, and stays open regardless of which library gets chosen. Worth revisiting once an editor library is actually selected, not before.
  - Exact ephemeral-draft cleanup mechanics (page-leave detection, etc.) — acknowledged as an implementation detail, not a design gap.
  - Who has resolve authority is still governed by the permission model in 4.7, which remains open on granularity.
  - What seeds the next fork once a pending PR resolves — do accumulated "unstaged" resolutions automatically open the next fork/PR cycle, or does someone have to explicitly start the next round?

### 4.5 Governance: Ruleset Convention

- Documents need a small, machine- and AI-readable rule set (e.g. "don't rewrite block IDs") so that agents working on a document locally don't make locally-reasonable, globally-disruptive changes.
- Modeled on the **LICENSE / SPDX-identifier pattern**: the actual ruleset text is published once, canonically, in the repo (like a LICENSE file). Each document's front matter carries a short pointer/version reference, not a copy of the full text — avoids drift between documents and a central rule change.
- **Risk:** a pointer only works if whatever touches the document actually reads and honors it. An AI agent unaware of Nog's conventions (e.g., a general-purpose coding assistant asked to "tidy up this file") has no reason to look it up.
  - Mitigation: back the convention with a **mechanical check** (pre-commit hook or CI validation) that flags disallowed changes to block IDs or front matter — treat the pointer as documentation for well-behaved agents, and the check as the actual enforcement for everyone else.
- Open question: single repo-wide ruleset (like one LICENSE) vs. per-folder/collection variation. Starting with one canonical ruleset is reasonable for a first version.

### 4.6 Scope Control

- Two distinct policies, not one:
  - **Default scope:** only markdown files are ever considered — a built-in default, not something that needs a config file to state.
  - **Exclusions:** specific markdown files/paths that exist in the repo but should never be touched by Nog (private drafts, internal-only notes). This does need a real ignore-file, reusing familiar `.gitignore` glob syntax.
- Consider supporting **both** a path-based ignore file (good for excluding whole folders) and a **front-matter flag** on individual files (good for one-off exceptions within an otherwise-included folder) — precedent: static site generators like Jekyll/Hugo support both mechanisms side by side.

<!-- nog:blk-008 -->
### 4.7 Permissions Model

- Google-Docs-style: read / comment / edit / own, applied at the Nog level.
- Authorized Nog users (with sufficient permission) can trigger committing a document back to the repo; this involves mapping the working representation back down to markdown with front matter.
- Open question: granularity of permissions — per document, per folder/collection, or repo-wide — not yet decided.

### 4.8 Read Surface: Reuse an Existing Static Site Generator

- The wider group's default, day-to-day experience of a document is not custom Nog interface — it's a rendered, navigable site built by an existing static site generator already common in the docs-as-code space, rebuilt from the repository on every commit.
- This was part of the original architecture sketch (the "GitBook-like" alternate floated early on) but hadn't been formally landed as a decision until now.
- Custom Nog interface work is reserved specifically for the moment someone actually wants to comment or edit — materializing the on-demand, block-ID-addressable view described in 4.2 — not for general reading or browsing.
- Consequence: Nog doesn't need to build or maintain its own renderer, navigation, or search. It only needs a thin commenting affordance layered on top of pages the static site generator already produces, plus the machinery in 4.2–4.7 for what happens once someone actually uses it.

## 5. Open Questions Carried Forward (not yet addressed by any decision above)

- **Scale of "wider group":** a handful of named external partners vs. an open-ended/public link changes whether lightweight, no-signup identification is enough or real accounts and moderation are needed.
- **Notification:** comment-and-response threads go cold without some way to tell participants when someone replies to or resolves their thread. Nothing yet defines this.
- **Provenance / rights:** once external, non-organisational people contribute wording that gets committed into the repo, there's a live question about attribution and consent for that contribution — not resolved, possibly fine to defer at small scale.
- **Moderation:** an externally-facing comment surface will eventually meet off-topic, hostile, or spam participants. Not yet addressed.
- **Ruleset scope:** single canonical ruleset vs. per-folder variants (see 4.5).
- **Permission granularity:** document-level vs. folder-level vs. repo-wide (see 4.7).

## 6. Naming

- Working name: **Nog** — "Notes On Git."
- Rationale: short, easy to say, playful register consistent with the original naming instinct, clean of any meaningful collision, direct acronym-to-function mapping.
- Rejected candidates, with reasons:
  - **Cog** ("Comments On Git") — collides with Cocogitto's `cog` CLI, a Conventional Commits/semver toolbox that operates directly on git repos (same domain, same likely audience). Also collides with at least three other unrelated existing tools using the same name (a Python code-generation tool, Replicate's ML container tool, Grafana's code-gen tool).
  - **Gloss** — thematically the strongest fit (a "gloss" is the historical term for marginal commentary on a text), but sits too close to Charm's *Glow* and *Lip Gloss* — markdown/git-terminal tools from the same maker, same audience, similar phonetics.
  - **Dog** — collides with an existing tool described as "a synchronisation tool that gathers changes in a central git repository" — too close to Nog's own domain.
  - **Ink** — collides with Ink & Switch, a local-first software research group whose work is directly adjacent to this exact problem space.
